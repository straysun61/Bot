但是目前遇到一个痛点任务：业务不仅需要处理传统的“带文字内容的电子 PDF”，还要能精准提取**手写的照片、纯图片以及全扫描件组成的 PDF**。为此，我们敲定了使用 **视觉大模型 (Vision LLM)** 来解析这些难以处理的手写画面，而不是在本地安装笨重的 Tesseract。

请你按照下列要求将代码系统升级：

### 任务 1：扩充依赖包 (Requirements)
为系统增加图像抓取与 PDF 转图片的工具。请在 `requirements.txt` 中增加：
- `PyMuPDF` (用于精准读取 PDF 及其图像层并可将其转换为图片资源)
- `Pillow` (用于处理图片并在需要时转为 Base64)

### 任务 2：扩展环境变量池
修改 `core/config.py`，允许配置调用 Vision API：
- 默认利用现有的 `OPENAI_API_KEY`。
- 新增 `VISION_LLM_MODEL: str = "gpt-4o"` 的配置项（因为该模型处理视觉多模态能力极强）。
- 扩展 `SUPPORTED_FILE_TYPES`，补充 `[".png", ".jpg", ".jpeg"]`，以原生支持用户直接上传图片文件。

### 任务 3：重构核心提取器 (DocumentParser)
修改 `core/rag_engine.py` 下的 `DocumentParser` 类中的方法逻辑。
需要你封装一个**调用 Vision LLM 提取文本的逻辑**：
1. **针对直接上传的图片 (`.png` 等)**：将图片转为 `base64` 编码。构造 OpenAI SDK 请求大模型接口 (`gpt-4o`)。System prompt 大致为：“_你是一个顶级的 OCR 与排版复原专家。请识别并提取出图片里的所有文本，特别是手写文字，如实复现，不要编造_”。把返回的结果作为提取出的 raw_text。
2. **针对原有的 PDF (`.pdf`)**：用 `PyMuPDF`(`fitz`) 拆解 PDF 页面。如果你使用普通的文本方法读不出任何实质性内容（即：字数异常少或者是全扫描图片），那么立即回退（Fallback）并将该页面转换为图像，复用上面写好的 Vision LLM 将图片识别并抽出该页手写字，拼接回原文。

### 任务 4：兼容现有接口处理流
确认在 `routers/documents.py` 中增加对新引入格式的支持。
保证不论是文档中的大量手写字被视觉大模型解译出后返回的字符串，还是纯文本字符串，随后都会无缝进入咱们之前的 `parent_splitter` 母子切割机里面，并乖乖存入 Chroma。

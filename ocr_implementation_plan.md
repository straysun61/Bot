# 文档 Bot 进阶改造计划：增强对“手写文本与图片扫描件”的支持

收到您的反馈！当前的 Bot 代码构架逻辑上是很完整的，但如您所说它目前是个局限于“数字内嵌文档”的处理器。使用了 `pdfplumber` 只能提取电脑生成的 PDF 文字。如果是图片扫描件、或者您的“手写文稿”，它会变成瞎子。

为了打破这层限制，让 Bot 能“看得见”手写文字并照常执行高维向量检索 (RAG)，我们需要在我们的服务层追加 **OCR (光学字符识别)** 能力或者 **视觉大模型 (VLM)** 能力。

## 升级提议选项设计 (Proposed Solutions)

给机器加上“看图片和手写字”的能力在目前的开源界或者 API 服务界各有不同成本和准确度。以下是可选的方案，**请您回复您更倾向使用哪种方案（或者结合）**，我将根据您的选择直接把代码写进 `core/rag_engine.py` 去！

### 方案 A：使用本地深度学习引擎 `PaddleOCR`
我们将全面改造当前的 `DocumentParser` 代码：
- 不再单一依赖 `pdfplumber` 拆字符。
- 如果检测到 PDF 中全是图片（比如手机拍的试卷或手写笔记），或者您直接上传了一张图片 (`.png`/`.jpg`格式)。
- 我们会在 Python 中调用 `PaddleOCR` 引擎，它针对基于中文的“手写中文”的识别率在全球范围内名列前茅。
- **劣势**: 这纯粹是免费的，但需要在您的本地启动环境里安装额外的底层库比如 `paddlepaddle`（百度的深度学习框架），配置相对繁琐一点点。

### 方案 B：调用多模态大视觉模型 (如 `GPT-4o` 或 `Qwen-VL`) 
这是当今最强最硬核的路线路线：
- 后台如果拿到难以解析的扫描手写 PDF，或者纯粹的手写拍照图。
- 我们直接使用 Base64 将图转化并发给第三方大模型平台的视觉 API（如智谱的 `GLM-4V` 或 OpenAI 的 `gpt-4o`）。
- 提示词设置为：“_请把这张图片里的手写中文内容一字不漏地全部复写为标准文字。_”
- **优势**: 对又乱又草的手写体的容错率高得惊人，不用费劲安装复杂的深度学习包本地跑。
- **劣势**: 会消耗在线大模型的 tokens（要花点钱）。
